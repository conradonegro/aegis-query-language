### BUG-4 (persistent): Implicit/comma JOIN still generated by LLM

**Questions:** q=1476, q=1486, q=1493 (3 Safety Violations, unchanged from prior run)

**Description:**
The prompt rule added in the previous iteration ("Never use comma-separated table lists in FROM clauses") has not eliminated the pattern. The LLM continues to generate queries like `FROM yearmonth, customers WHERE yearmonth.customerid = customers.customerid` for certain multi-table queries, which the safety engine correctly rejects.

Example error:
```
Safety Violation: Implicit or cross JOIN detected: every JOIN must have an explicit ON or USING condition.
```

**Proposed fix:**
The prompt rule alone is insufficient. Add a pre-safety translation step in the compiler pipeline that rewrites comma-separated FROM clauses into explicit `INNER JOIN ‚Ä¶ ON` syntax before the safety engine sees the query. This makes the fix deterministic instead of relying on LLM compliance.

Alternatively, strengthen the prompt rule with a concrete counter-example pair:
> "WRONG: `FROM yearmonth, customers WHERE yearmonth.customerid = customers.customerid`
> CORRECT: `FROM yearmonth JOIN customers ON yearmonth.customerid = customers.customerid`"

### BUG-5 (new): CTE-prefixed output column alias not bypassed in translator

**Questions:** q=1479, q=1480 (2 Translation Errors)

**Description:**
BUG-3 fixed bare column references to CTE output aliases (e.g., bare `total_consumption` in ORDER BY). However, PROMPT-3 (unconditional column qualification) now causes the LLM to emit the CTE alias as a prefix ‚Äî e.g., `agg.total_consumption` instead of bare `total_consumption`.

When the LLM writes:
```sql
WITH agg AS (
    SELECT SUBSTRING(date, 1, 4) AS year, SUM(consumption) AS total_consumption
    FROM yearmonth GROUP BY year
)
SELECT agg.year FROM agg ORDER BY agg.total_consumption DESC LIMIT 1
```

The outer `agg.total_consumption` goes through `_resolve_column_with_prefix`. This path looks up `agg` as a schema table alias ‚Äî it is not found, and the translation fails:
```
Translation Error: Column 'total_consumption' does not exist in the schema context.
```

The BUG-3 fix only patched `_resolve_column_without_prefix`. PROMPT-3 shifted the failure from the bare-column path to the prefixed-column path.

**Proposed fix:**
In `_resolve_column_with_prefix`, before looking up the prefix in the physical schema, check whether the prefix matches a CTE name. Collect CTE names via the existing `_collect_cte_aliases` set (already computed in `translate()`). If the prefix is a CTE name, bypass physical resolution and return without rewriting the node:

```python
def _resolve_column_with_prefix(self, node, tbl_name, c_name, maps, scope,
                                 column_datatypes, cte_aliases, cte_col_aliases):
    if tbl_name in cte_aliases:
        return  # CTE table reference ‚Äî no physical resolution needed
    ...
```

This pairs naturally with the existing BUG-3 fix for bare CTE column aliases.

### BUG-6 (new): EXTRACT on CAST expression rejected by safety engine

**Questions:** q=1340 (1 Safety Violation)

**Description:**
The safety engine rejects `EXTRACT(YEAR FROM CAST(expense_date AS DATE))` with:
```
Safety Violation: EXTRACT numeric target must be natively bound to a column, found 'Cast'.
```

The safety engine's EXTRACT validation requires the argument to be a bare column reference (`exp.Column`). However, `EXTRACT(YEAR FROM CAST(col AS DATE))` is valid PostgreSQL and is a natural pattern when a column stores dates as TEXT and needs explicit casting before temporal extraction.

**Proposed fix:**
In `safety.py`, extend the EXTRACT argument validation to also permit `exp.Cast` nodes (in addition to `exp.Column`). Optionally, also permit `exp.Anonymous` date functions like `TO_DATE(...)`. The key invariant to preserve is that function calls inside EXTRACT are controlled ‚Äî only `CAST` and `TO_DATE`/`TO_TIMESTAMP` are safe to add.

```python
EXTRACT_ALLOWED_ARG_TYPES = (exp.Column, exp.Cast)

def _validate_extract(self, node: exp.Extract) -> None:
    arg = node.this  # the temporal expression
    if not isinstance(arg, EXTRACT_ALLOWED_ARG_TYPES):
        raise SafetyViolationError(
            f"EXTRACT numeric target must be natively bound to a column or CAST, "
            f"found '{type(arg).__name__}'"
        )
```

## Prompt failures (wrong refusals caused by LLM reasoning)

### PROMPT-1 (persistent): Over-refusal based on sample values ‚Äî q=1484, q=1331, q=1356

**Description:**
Despite rule 10 being extended to state that sample values are not exhaustive, the LLM continues to refuse queries when the queried value is not in the displayed samples. Three distinct cases:

**q=1484** ‚Äî "How many more 'discount' gas stations does Czech Republic have vs Slovakia?"
The LLM sees `gasstations.segment` samples: `Other, Premium, Noname`. It refuses: *"there is no 'discount' segment value in the provided schema context."* The actual segment values include Discount ‚Äî it simply wasn't in the top-3 by frequency. The LLM is violating rule 10.

**q=1331** ‚Äî "What is the amount of the funds that the Vice President received?"
The LLM sees `member.position` samples: `Member, Inactive, Secretary`. It refuses: *"The schema does not contain a 'Vice President' position value."* Vice President exists in the data; it is just not a top-3 value.

**q=1356** ‚Äî "Which department was the President of the club in?"
Same pattern. `position` sample values don't include `President`, so the LLM refuses.

**Proposed fix:**
Rule 10 needs a stronger, more direct phrasing. The current text uses "never infer that a value is absent." The LLM interprets this as permission to refuse if it cannot *confirm* a value exists. Reframe as a strict mandate:

> "10. ‚Ä¶ **You MUST attempt a query if the column exists in the schema, even if the queried value is not in the sample values. Sample values are frequency-based illustrations only ‚Äî they are never exhaustive. Refusing because a value does not appear in the samples is a violation of this rule.**"

Additionally, consider whether showing sample values for low-cardinality columns like `segment` (4‚Äì5 values) and `position` (~8‚Äì10 values) should show all values instead of a 3-item sample. A threshold could be added to `discover_metadata.py`: if `COUNT(DISTINCT col) <= 8`, show all values instead of the top 3.

### PROMPT-3 (partially fixed): Ambiguous naked column survives ‚Äî q=1526

**Description:**
PROMPT-3 added an unconditional column qualification rule. For q=1526, the LLM generates a complex CTE that joins `transactions_1k` and `yearmonth` internally, and still emits a bare `customerid` in the outer query:

```
Translation Error: Ambiguous naked column 'customerid'. Belongs to multiple scoped tables: ['yearmonth', 'transactions_1k'].
```

The rule exists but the LLM fails to comply in deeply nested CTEs. This may require a translator-level fallback: if a bare column is unambiguous in the outer scope (after CTE columns are resolved), resolve it without raising, similar to how CTE output aliases are bypassed.

### PROMPT-5 (new): Legitimate refusals blocking answerable questions ‚Äî q=1515, q=1362

**Description:**
Two refusals that appear legitimate on the surface but may be answerable:

**q=1515** ‚Äî "What segment did the customer have at 2012/8/23 21:20:00?"
The LLM refuses because `customers.segment` has no timestamp dimension. The gold SQL appears to join `transactions_1k` on the specific date/time to find the customer, then return their static segment. The LLM correctly identifies the schema limitation (segment is not time-versioned) but incorrectly concludes the query is unanswerable ‚Äî the gold just returns the customer's current static segment for the matching transaction row.

**q=1362** ‚Äî "How many cities are there in Orange County, Virginia?"
The LLM refuses because it cannot confirm 'Orange County' exists in the data. The `zip_code` table has a `county` column. This is a PROMPT-1-class refusal (value not in samples) rather than a schema absence. Should be treated the same as PROMPT-1.

### STRATEGY-1 (persistent): MIN(row) vs MIN(SUM per customer) ‚Äî q=1472

LLM finds the customer with the minimum individual `consumption` row in 2012. Gold groups by customerid, sums all 2012 consumption rows, then finds the minimum total. The LLM misses the GROUP BY + SUM aggregation step.

### STRATEGY-2 (persistent): AVG(rows) vs SUM/12 ‚Äî q=1473

LLM generates `AVG(consumption)` across all yearmonth rows for SME in 2013. Gold generates `SUM(consumption) / 12`. The question asks for "average monthly consumption" which means total divided by 12 months, not average per row. Known from prior run.

### STRATEGY-3 (persistent): MAX(row) vs SUM(month) ‚Äî q=1498

LLM generates `MAX(consumption)`. Gold generates `SUM(consumption) GROUP BY month ORDER BY SUM DESC LIMIT 1`. Known from prior run.

### STRATEGY-4 (persistent): COUNT(CASE) per row vs SUM(CASE) per customer ‚Äî q=1490, q=1505

q=1490: LLM uses `COUNT(CASE WHEN consumption > 46.73)` over all rows, denominator is all rows. Gold uses `SUM(CASE‚Ä¶)/COUNT(customerid)` ‚Äî per-customer denominator. Known from prior run.

q=1505: LLM uses `COUNT(DISTINCT customerid)` (unique customers with any month > 1000). Gold uses `COUNT(*)` counting all matching customer-month rows. Known from prior run.

### STRATEGY-5 (persistent): Direct date filter on transactions_1k vs yearmonth join ‚Äî q=1500, q=1501, q=1529

q=1500, q=1501: LLM extracts month/year directly from `transactions_1k.date` instead of joining `yearmonth` and filtering on `yearmonth.date = 'YYYYMM'`. Gold uses the `yearmonth` billing-period table for month-level filtering. Known from prior run.

q=1529: LLM uses `EXTRACT(YEAR/MONTH FROM transactions_1k.date)`. Gold joins `yearmonth` for month = '201201'. Same root cause.

### STRATEGY-6 (new): Complex multi-segment comparison ‚Äî q=1481

Gold uses a flat `SUM(CASE WHEN segment='X' THEN consumption END) / COUNT(customerid)` pattern across all joined rows. LLM builds elaborate per-customer CTEs and ends up with a different computation unit (per-customer minimum vs per-row average). The gold SQL's structure is unusual and difficult to infer from the question alone.

### STRATEGY-7 (new): "N times more" = ratio, not count ‚Äî q=1359

Question asks "how many times was budget for 'Yearly Kickoff' more than 'October Meeting'?" LLM returns a comparison of counts (`COUNT(CASE WHEN amount > threshold)`). Gold returns a ratio: `SUM(Yearly Kickoff amount) / SUM(October Meeting amount)`. "How many times more" is a division, not a count.

### STRATEGY-9 (new): "Top spending customer" ‚Äî different ranking base ‚Äî q=1531

LLM ranks by total `SUM(transactions_1k.price)`. Gold ranks by `yearmonth.consumption` (the billing-period consumption figure). Both are reasonable interpretations of "top spending" ‚Äî they disagree on which table is authoritative for spending.

### STRATEGY-10 (new): Percentage formula ‚Äî q=1525, q=1528

q=1525: LLM uses `COUNT(DISTINCT customerid WHERE currency='EUR') / COUNT(DISTINCT customerid)`. Gold uses `SUM(CASE WHEN currency='EUR')` / `COUNT(customerid)` ‚Äî counts non-distinct rows. Results differ when a customer appears in multiple transaction rows.

q=1528: LLM adds `WHERE country='SVK'` then computes `COUNT(CASE WHEN segment='Premium')/COUNT(*)`. Arithmetic is correct but result may differ from gold `SUM(CASE WHEN SVK AND Premium)/SUM(CASE WHEN SVK)` due to parameter binding (`:p2 = 'Premium'` vs gold's literal `'Premium'` ‚Äî case sensitivity possible).

### STRATEGY-11 (new): Event filter ‚Äî q=1322, q=1338

q=1322: LLM HAVING clause is correct but it then wraps into a COUNT outer query that double-counts. Gold does a single `GROUP BY event_id HAVING COUNT > 10` then counts those events.

q=1338: LLM returns all expense columns. Gold returns only `approved`. Also, date comparison format: LLM uses exact date string, gold uses `event_date LIKE '2019-10-08%'`.
