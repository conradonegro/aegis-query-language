## Bugs (execution-layer failures)

---

### BUG-1: Date string parameters crash asyncpg with `toordinal` error

**Questions:** q=1509, q=1514, q=1521, q=1525, q=1529 (5 Internal Server Errors)

**Description:**
When the LLM generates a query comparing against a date or timestamp literal (e.g. `WHERE t.date > '2012/1/1'` or `WHERE t.date = '2012/8/24'`), the translator parameterizes the string literal as `$1`. asyncpg then infers the bind parameter type as `DATE` or `TIMESTAMP` from the column context and attempts to call `.toordinal()` on the Python string, crashing the ASGI application with:

```
AttributeError: 'str' object has no attribute 'toordinal'
```

This is the same class of problem as BUG Fix 1 (integer literals). The `transactions_1k` table has `date DATE` and `time TIME` columns. Any query that filters on these columns with a parameterized literal crashes.

Example failing query (q=1509):
```sql
SELECT COUNT(*) FROM transactions_1k AS t
JOIN gasstations AS g ON t.gasstationid = g.gasstationid
WHERE g.country = :p1 AND t.date > :p2   -- :p2 = '2012/1/1', fails as DATE
```

**Proposed fix:**
Extend `_parameterize_literals` in `translator.py` to skip string literals that look like date or time values — the same approach as Fix 1 for integers. Use the `col_datatypes` map already collected during tree walking to detect when a literal is being compared against a date/time column, or simply detect date-like string patterns (e.g. matching `\d{4}[-/]\d{1,2}`) and leave them inline.

An alternative is to also add a prompt rule telling the LLM to use `CAST('...' AS DATE)` for date comparisons, but the root cause is in the translator and should be fixed there.

### BUG-2: UNION blocked at safety engine root check

**Questions:** q=1482

**Description:**
When the LLM generates a UNION query to answer "biggest AND lowest" comparisons in a single result set, the safety engine rejects it with:

```
Safety Violation: Root node must be SELECT, found Union
```

`exp.Union` was added to the ALLOW_LIST as part of a previous fix, but the root node check in the safety engine explicitly asserts `isinstance(root, exp.Select)` before the allow-list walk. This check runs first and rejects UNION regardless of the allow-list.

Example LLM query (q=1482):
```sql
WITH consumption AS (...)
SELECT 'highest' AS label, segment FROM consumption ORDER BY pct_increase DESC LIMIT 1
UNION ALL
SELECT 'lowest' AS label, segment FROM consumption ORDER BY pct_increase ASC LIMIT 1
```

**Proposed fix:**
In `safety.py`, change the root node assertion to accept both `exp.Select` and `exp.Union`:
```python
if not isinstance(root, (exp.Select, exp.Union)):
    raise SafetyViolationError("Root node must be SELECT or UNION")
```
Then recursively validate each branch of the UNION (both `this` and `expression` of `exp.Union`) through the normal allow-list walk.

### BUG-3: CTE output column aliases not bypassed in translator

**Questions:** q=1479, q=1480, q=1486, q=1531 (4 Translation Errors)

**Description:**
Fix 3 taught the translator to bypass CTE table aliases when resolving `FROM` targets and `JOIN ON` conditions. It did not handle the case where the outer query references column aliases **declared inside a CTE's SELECT clause**.

When the LLM writes:
```sql
WITH agg AS (
    SELECT SUBSTRING(date, 1, 4) AS year, SUM(consumption) AS total_consumption
    FROM yearmonth GROUP BY year
)
SELECT year FROM agg ORDER BY total_consumption DESC LIMIT 1
```

The outer `ORDER BY total_consumption` hits `_resolve_column_with_prefix`. The translator looks up `total_consumption` in the schema and fails because it is not a physical schema column — it is a CTE-derived alias declared with `AS`.

This affects any query where an outer SELECT, WHERE, HAVING, ORDER BY, or GROUP BY references a column name produced by a CTE. All four affected questions use aggregation CTEs (`SUM(...) AS total_consumption`, `COUNT(...) AS customer_count`, `SUM(...) AS total_spending`).

**Proposed fix:**
In `DeterministicTranslator`, extend `_collect_cte_aliases` to also collect the **output column aliases** of each CTE (not just the CTE name itself). Walk each CTE's SELECT expression list, extract any `AS alias` declarations, and add them to a `cte_column_aliases: set[str]` set. Then in `_resolve_column_with_prefix` (and the bare-column resolution path), if the column name is in `cte_column_aliases`, return early without trying to resolve it against the schema.

```python
@staticmethod
def _collect_cte_column_aliases(tree: exp.Expression) -> set[str]:
    aliases: set[str] = set()
    for cte in tree.find_all(exp.CTE):
        for sel in cte.find_all(exp.Alias):
            aliases.add(sel.alias.lower())
    return aliases
```

### BUG-4: Implicit/comma JOIN generated and blocked by safety engine

**Questions:** q=1493

**Description:**
Despite rule 7 permitting only explicit JOINs, the LLM occasionally generates old-style implicit joins using comma-separated FROM clauses (e.g., `FROM yearmonth, customers WHERE yearmonth.customerid = customers.customerid`). The safety engine correctly rejects these, but the error should never reach the safety engine — it should be prevented at the prompt level.

Example (q=1493):
```
Safety Violation: Implicit or cross JOIN detected: every JOIN must have an explicit ON or USING condition.
```

**Proposed fix:**
Strengthen rule 7 in `system.jinja` with an explicit prohibition:
> "Never use comma-separated table lists in FROM clauses (e.g. `FROM a, b`). Always use explicit `JOIN ... ON` syntax. This applies even when the join condition is simple."

## Prompt failures (wrong refusals or wrong answers caused by LLM reasoning)

---

### PROMPT-1: Sample values cause over-confident refusals

**Questions:** q=1481, q=1528

**Description:**
Fix 6 added sample values to the prompt to help the LLM understand column contents. This works well when the query value **is** in the samples (q=1484 was fixed). However, when a queried value is **not** in the samples, the LLM incorrectly infers the value does not exist in the data and refuses — directly contradicting rule 10.

q=1481 refusal: LLM sees sample values `201112, 201201, 201202` for `yearmonth.date` and concludes: *"date field shows values 201112-201202 which does not extend to 2013 data based on the samples provided."* In reality 2013 data exists — the samples are the three lexicographically-smallest values.

q=1528 refusal: LLM sees `gasstations.segment` samples `Discount, Noname, Other` and refuses: *"'premium' is not in the sample values."*

**Proposed fix:**
Add a rule after the sample values in the schema block, or extend rule 10:
> "Sample values are illustrative examples only — they are not exhaustive. Never infer that a value is absent from the data based on the samples shown. Rule 10 still applies: always attempt the query if the schema contains the relevant column."

Consider also whether the 3-sample limit should be raised for low-cardinality categorical columns (`gasstations.segment` has only ~4 distinct values — showing all of them would be more useful than showing 3 of 4).

### PROMPT-2: Database context name confused with table name

**Questions:** q=1317

**Description:**
The LLM refuses because it cannot find a table named "Student_Club". The question asks: *"Among the students from the Student_Club who attended the event 'Women's Soccer'…"*. The LLM interprets "Student_Club" as a required table name rather than the name of the system/database being queried.

The student_club schema has `member`, `event`, `attendance` etc. — the "Student_Club" IS the system. This pattern will occur in any BIRD database where the database name appears verbatim in a question.

**Proposed fix:**
Pass the `source_database` name (the BIRD database identifier) into the prompt and add it to the schema header:
> "You are querying the **student_club** database. Questions may refer to entities by the database name (e.g., 'Student_Club members' means all rows in the `member` table)."

This requires threading the database name through the compiler pipeline from `source_database` on `AbstractTableDef` (already present) up to the `PromptEnvelope`.

### PROMPT-3: Ambiguous bare column survives in complex CTE+JOIN queries

**Questions:** q=1526

**Description:**
Fix 4 added the rule to always qualify column references with table aliases. This works for simple queries but q=1526 still produces:

```
Translation Error: Ambiguous naked column 'customerid'. Belongs to multiple scoped tables: ['transactions_1k', 'yearmonth'].
```

The query joins `transactions_1k` and `yearmonth` inside a CTE, and the outer query references `customerid` without qualification. The rule in the prompt says to prefix columns "when the query involves more than one table" but in a complex nested CTE the LLM misses one reference.

**Proposed fix:**
Harden the rule from "when the query involves more than one table" to an unconditional mandate:
> "Always qualify EVERY column reference with its table or CTE alias — in SELECT, FROM, WHERE, ON, HAVING, GROUP BY, and ORDER BY clauses. There are no exceptions, even for seemingly unambiguous single-column names."

### PROMPT-4: EXTRACT used on TEXT date columns in transactions_1k

**Questions:** q=1500, q=1501 (incorrect answers)

**Description:**
Questions asking "which products/stations appeared in September 2013" produce incorrect results. The gold SQL joins `transactions_1k` with `yearmonth` and filters on `yearmonth.date = '201309'` (the YYYYMM format). The LLM instead applies `EXTRACT(YEAR FROM t.date)` and `EXTRACT(MONTH FROM t.date)` directly on `transactions_1k.date`.

Two sub-problems:
1. `transactions_1k.date` is likely type `DATE` (EXTRACT would work syntactically) but the BIRD gold standard uses the `yearmonth` table for month/year-level filtering — meaning `transactions_1k.date` is a transaction-level timestamp while `yearmonth.date` is the coarser billing period.
2. The LLM does not understand the role of `yearmonth` as a period-aggregation table.

**Proposed fix:**
This is partially a description issue. Improving the description for `yearmonth` (e.g., "monthly billing period table; each row represents one customer's aggregate consumption for one calendar month in YYYYMM format") and `transactions_1k` (e.g., "individual gas station transactions with transaction-level date and time") would help the LLM understand when to join via `yearmonth` for period filtering. This is a metadata quality improvement, not a code change.

## Infrastructure

### INFRA-1: Anthropic API rate limiting (429) wipes entire database's results

**Questions:** q=1331–q=1368 (15 student_club questions — all 429 errors)

**Description:**
Running 50 questions at concurrency=5 exhausted Haiku's rate limits partway through the student_club section. The LLM gateway does not retry on 429 — it immediately propagates the error. All 15 consecutive student_club questions from q=1331 onward failed, reducing accuracy by ~30 percentage points on student_club (5.0% instead of a potential ~35%).

**Proposed fix:**
Add exponential backoff with jitter and a configurable retry count (e.g. 3 retries, base delay 2s) in the LLM gateway's HTTP call layer. The 429 response includes `Retry-After` header which should be respected. Alternatively, reduce benchmark concurrency to 2 for multi-database runs.

## Wrong answers — query strategy mismatches

These affect 11 questions that execute without error but return incorrect results. They are harder to fix with prompt rules and may require better column descriptions or semantic reasoning.

---

### STRATEGY-1: FK traversal — LLM returns ID instead of human-readable join

**Questions:** q=1312 (`Angela Sanders's major`)

LLM returns `member.link_to_major` (a UUID FK) instead of joining to `major` table to return `major.major_name`. The FK relationship is declared in the schema.

Root cause: the column description for `link_to_major` does not say it is a FK to `major.major_name`. Improving the column description ("foreign key to major.major_id — join to get major_name") would guide the LLM. This is a metadata quality issue.

---

### STRATEGY-2: "Highest monthly consumption" semantic ambiguity

**Questions:** q=1498

LLM generates `MAX(consumption)` (max of individual rows). Gold generates `SUM(consumption) GROUP BY month ORDER BY SUM DESC LIMIT 1` (max total for a calendar month). "Highest monthly consumption" means the month with the greatest total — not the single highest row. Hard to fix with prompt rules; would benefit from clearer column descriptions.

---

### STRATEGY-3: "Average monthly" = total/12, not AVG over rows

**Questions:** q=1473

LLM computes `AVG(consumption)` across all rows for SME in 2013. Gold computes `SUM(consumption) / 12`. For a question about "average monthly consumption over a year", the correct interpretation is total divided by 12 months. `AVG` gives average per consumption row (which can include partial months). Difficult to address without explicit domain knowledge in descriptions.

---

### STRATEGY-4: SUM(CASE) pattern vs COUNT DISTINCT for conditional counting

**Questions:** q=1490, q=1505

q=1490: LLM generates `COUNT(CASE WHEN x > N THEN 1 END) / COUNT(*)` (percentage of rows). Gold uses `SUM(CASE...) / COUNT(customerid)` — counting per customer, not per row. Subtle difference when customers have multiple yearmonth rows.

q=1505: LLM generates `COUNT(DISTINCT customerid)` (unique customers with any month > 1000). Gold uses `COUNT(*)` which counts customer-month pairs. The question "how many customers have monthly consumption over 1000" is ambiguous; gold apparently counts rows.

These are semantic ambiguities in the benchmark questions themselves. Hard to fix systematically.

---

### STRATEGY-5: yearmonth join pattern for transaction month filtering

**Questions:** q=1500, q=1501, q=1533

Covered in PROMPT-4 above. Queries need to join `transactions_1k → yearmonth` for month-level filtering rather than extracting directly from `transactions_1k.date`. Column description improvements in metadata would be the right fix.
