### BUG-001 · CRITICAL · Leaked API Key in `.env`

**File:** `.env`, line 15
**Symptom:** The `.env` file on disk contains a key in the format `sk-proj-lQ2JEJ...` (a real OpenAI project key). Even though `.gitignore` excludes `.env`, the key is on disk and a single accidental `git add .env` or mis-configured CI would expose it. If this key has ever been used in a pushed commit, it is already compromised.
**Fix:** Rotate the key at OpenAI immediately. Replace the value in `.env` with a placeholder. Commit an `.env.example` with all placeholder values so new contributors know what to populate.

### BUG-002 · CRITICAL · `test_compiler_engine.py` References Non-Existent `_session_store` Attribute

**File:** `tests/test_compiler_engine.py`, lines ~138, 139, 188, 189, 205, 206
**Symptom:** Tests directly access `compiler_engine._session_store`. The `CompilerEngine` class (`app/compiler/engine.py` line 56) stores the session store as `self.session_store` (public, no underscore). The internal state dict of `SessionStore` is `_local`. These attribute accesses raise `AttributeError` at runtime, causing multiple test cases to fail silently or with errors rather than assertion failures.
**Fix:** Replace `compiler_engine._session_store` with `compiler_engine.session_store._local` (in-memory fallback) or add a `get_raw_session` test-helper method to `SessionStore`.

### BUG-003 · CRITICAL · `test_ollama.py` and `test_prompting_gateway.py` Construct `PromptEnvelope` with Invalid Fields

**Files:**
- `tests/test_ollama.py`, lines ~14–18
- `tests/test_prompting_gateway.py`, lines ~44–49

**Symptom:** Both files construct `PromptEnvelope` passing fields `schema_context` and `hints` that do not exist on the model. `PromptEnvelope` (defined in `app/compiler/models.py`) only has `system_instruction`, `user_prompt`, and `chat_history`. Pydantic will raise `ValidationError` on model construction, causing these tests to fail before the first assertion.
**Fix:** Remove the spurious fields. If `schema_context` and hints need to be represented in tests, embed them in the `system_instruction` string or update `PromptEnvelope` to include them (but note this would require all callers to be updated).

### BUG-004 · HIGH · LLM Refusal Detection Broken for All Real Providers

**Files:** `app/compiler/ollama.py`, `app/compiler/openai_gateway.py`, `app/compiler/anthropic_gateway.py`, `app/compiler/google_gateway.py`, `app/compiler/xai_gateway.py`, `app/compiler/engine.py`
**Symptom:** All real LLM gateways pre-parse the LLM JSON response and return only `parsed["sql"]` as `LLMResult.raw_text`. When the LLM outputs a refusal (`{"refused": true, "reason": "..."}`), the gateway attempts `parsed["sql"]` — which will `KeyError` or return `None` — rather than preserving the full JSON for the engine to inspect. The `CompilerEngine.compile()` refusal check (engine line ~155–183) reads `raw_text`, but by the time it does, the refusal signal has been stripped.

Specifically:
- If `parsed["sql"]` raises `KeyError`, the gateway raises `LLMGenerationError` (Ollama) or an uncaught `KeyError` (others) — the refusal reason is lost.
- If `parsed.get("sql")` returns `None`, the engine receives `None` as the SQL string and crashes in the parser rather than raising a clean `LLMRefusalError`.

The mock gateway is unaffected because it returns the full JSON object.
**Fix:** All gateways should return the full raw JSON string as `LLMResult.raw_text` without pre-extracting the SQL. The engine's existing JSON parsing + refusal check logic was designed for this. Remove the `parsed["sql"]` extraction from the gateways and let the engine handle it uniformly. If the field is missing (malformed LLM output), that is an `LLMGenerationError`.

### BUG-005 · HIGH · `MockLLMGateway` Returns JSON; Real Gateways Return Plain SQL

**File:** `app/compiler/gateway.py`
**Symptom:** `MockLLMGateway.generate()` returns `json.dumps({"sql": self.mock_response_sql})`, a JSON string. All real gateways return plain SQL. This means tests exercise the `json.loads` success path in `CompilerEngine.compile()`, while production exercises the regex fallback path or the raw-SQL path. The refusal-detection code path is only exercised in tests, never in production. The two code paths produce different intermediate states during compilation.
**Fix:** This is a consequence of BUG-004. Fixing BUG-004 (all gateways return full JSON) makes the mock and real gateways consistent.

### BUG-006 · HIGH · `SELECT FOR UPDATE` Used Against SQLite in Tests

**File:** `app/api/router.py`, lines ~130–132 and ~229–231
**Symptom:** The `/generate` and `/execute` handlers call `select(ChatSession).where(...).with_for_update()` to lock the session row. SQLite does not support `SELECT ... FOR UPDATE`. Tests that exercise these routes with the SQLite in-memory backend will fail with a `CompileError: FOR UPDATE is not supported for SQLite`.
**Fix:** Use a conditional: emit `with_for_update()` only when not running against SQLite (check dialect), or use an application-level lock (e.g., `asyncio.Lock` keyed by session ID) that works across backends.

### BUG-007 · HIGH · `ExecutionContext` Passed a Non-Existent `metadata` Field

**File:** `app/api/router.py`, lines ~266–269
**Symptom:**
```python
context = ExecutionContext(
    tenant_id="default_tenant",
    user_id="api_user",
    metadata={}
)
```
`ExecutionContext` (defined in `app/execution/models.py`) does not have a `metadata` field. In Pydantic v2, extra fields are ignored by default, so this does not raise — but it signals a disconnection between what the router intends to pass and what `ExecutionContext` accepts. If `model_config` is tightened to `extra="forbid"` (good practice for security), this will break.
**Fix:** Remove `metadata={}` from the constructor call, or add a `metadata: dict = field(default_factory=dict)` to `ExecutionContext` if the field is intentional.

### BUG-008 · MEDIUM · `datetime.utcnow` (Deprecated) Used in ORM Defaults

**File:** `app/api/meta_models.py`, multiple column defaults
**Symptom:** Several `Column(DateTime, default=datetime.utcnow)` usages. `datetime.utcnow()` is deprecated in Python 3.12 and returns a naive datetime, which causes `DeprecationWarning` at runtime and may produce incorrect behavior when comparing with timezone-aware datetimes.
**Fix:** Replace `datetime.utcnow` with `lambda: datetime.now(timezone.utc)` for all column defaults. SQLAlchemy also accepts `server_default=func.now()` for server-side timestamping which avoids the issue entirely.

### BUG-009 · MEDIUM · Redis URL May Contain Credentials Logged in Plaintext

**File:** `app/main.py`, line ~125
**Symptom:**
```python
logger.info(f"Session store: Redis ({redis_url})")
```
If `redis_url` is of the form `redis://:password@host:6379/0`, the password is written to logs.
**Fix:** Parse the URL and redact the password before logging: use `urllib.parse.urlparse`, replace `.password` with `***`, and reconstruct.

### BUG-010 · MEDIUM · `OllamaLLMGateway` (and All Real Gateways) Inherit from `Protocol`

**Files:** `app/compiler/ollama.py`, `app/compiler/openai_gateway.py`, `app/compiler/anthropic_gateway.py`, `app/compiler/google_gateway.py`, `app/compiler/xai_gateway.py`
**Symptom:** All gateways are declared as `class XxxGateway(LLMGatewayProtocol):`. Inheriting from a `Protocol` class in Python means the class itself becomes a protocol subtype, not a concrete implementation. `mypy` in strict mode may accept this, but the semantics are wrong — concrete classes should implement protocols structurally (duck typing), not inherit from them. This can also suppress useful `mypy` errors (e.g., if a method is missing, inheriting from the protocol provides a no-op instead of an error).
**Fix:** Remove the `(LLMGatewayProtocol)` inheritance from all gateway classes. Use `isinstance` checks against the Protocol if runtime type checking is needed, or add `@runtime_checkable` decorator and use structural typing.

### BUG-011 · MEDIUM · `TranslatorProtocol` Signature Does Not Match `DeterministicTranslator`

**File:** `app/compiler/interfaces.py`, `app/compiler/translator.py`
**Symptom:** `TranslatorProtocol.translate` is declared as:
```python
def translate(self, ast: ValidatedAST, schema: RegistrySchema) -> ExecutableQuery: ...
```
But `DeterministicTranslator.translate` accepts additional parameters: `abstract_query_hash`, `safety_version`, `row_limit`, `relationships`. The protocol is too narrow; code relying on the protocol type to call `translate` would fail to pass the required arguments.
**Fix:** Update `TranslatorProtocol.translate` to include all parameters that `DeterministicTranslator.translate` requires. Use keyword-only arguments with defaults where appropriate.

## Part 2 — Improvement Suggestions

Suggestions are classified as **SECURITY**, **CORRECTNESS**, **PERFORMANCE**, **MAINTAINABILITY**, or **OBSERVABILITY**.

---

### IMP-001 · SECURITY · Add Authentication and Authorization

**Current state:** Zero auth. All requests are accepted; `tenant_id` and `user_id` are hardcoded strings.
**Suggestion:** At minimum, implement API-key-per-tenant authentication:
1. Add a `TenantCredential` table with hashed API keys.
2. Add a FastAPI `Security` dependency that validates the `Authorization: Bearer <key>` header, resolves `tenant_id`, and injects it into the request.
3. Remove the hardcoded `"default_tenant"` and `"api_user"` strings from the router.
4. Gate the Steward UI (metadata endpoints) behind an admin role.

A more complete solution would use JWTs with claims for `tenant_id`, `user_id`, and scopes.

---

### IMP-002 · SECURITY · Protect the Steward UI

**Current state:** Metadata endpoints (`/api/v1/metadata/*`) and the Steward UI (`/`) require no authentication. Anyone who can reach the service can redefine the schema, promoting versions that change what queries the LLM can generate.
**Suggestion:** All metadata-mutating endpoints should require an `admin` scope. The static UI should require session-based auth. Until IMP-001 is implemented, consider at minimum adding an IP allowlist for the metadata endpoints.

---

### IMP-003 · PERFORMANCE · Share `httpx.AsyncClient` Instances in LLM Gateways

**Files:** All LLM gateway files
**Current state:** Each `generate()` call opens and closes an `httpx.AsyncClient`, creating and tearing down TCP connections for every LLM request.
**Suggestion:** Move the `httpx.AsyncClient` to a class-level instance (or use the `lifespan` context manager in `main.py` to create and close it). Use `httpx.AsyncClient(limits=httpx.Limits(max_connections=20), timeout=60.0)`. This enables HTTP/1.1 keep-alive and HTTP/2 multiplexing, reducing latency by 50–200ms per request.

---

### IMP-004 · PERFORMANCE / RELIABILITY · Populate RAG Store from Registry Artifact

**Current state:** `InMemoryVectorStore` is never populated in the production code path. The RAG store exists but has no data source.
**Suggestion:** After `RegistryLoader.load_active_schema()` succeeds, iterate the compiled registry artifact and populate `InMemoryVectorStore` with any categorical values defined in column metadata. Add a `categorical_values: list[str]` field to `AbstractColumnDef` (or load them from a separate table). Also reload the RAG store when a new schema version is activated.

---

### IMP-005 · RELIABILITY · Make Redis Failure Non-Silent

**File:** `app/compiler/session_store.py`
**Current state:** If Redis is unreachable, `SessionStore` falls back to an in-memory `dict` without logging an error or warning. In multi-worker deployments this silently breaks follow-up query continuity.
**Suggestion:** Emit a `logger.warning("Redis unavailable, falling back to in-memory session store — multi-worker sessions will not be consistent")` on fallback. Consider raising in `ENVIRONMENT=production` rather than silently degrading.

---

### IMP-006 · RELIABILITY · Add Retry Logic for LLM Gateway Calls

**Files:** All LLM gateway files
**Current state:** LLM API failures propagate immediately as exceptions.
**Suggestion:** Add exponential backoff retry (e.g., via `tenacity`) for transient errors (HTTP 429, 500, 503, timeout). Configuration: 3 retries, base delay 0.5s, max delay 4s, jitter. Log each retry attempt. For 429 (rate limit), use the `Retry-After` header if present.

### IMP-007 · RELIABILITY · Handle LLM Output Parse Failures Gracefully

**File:** `app/compiler/engine.py`
**Current state:** If the LLM outputs neither valid JSON nor parseable SQL, the engine raises an unhandled exception that surfaces as a 500.
**Suggestion:** Add a dedicated `LLMOutputParseError` exception class. Catch the case where both `json.loads` and regex extraction fail, and raise this error with the raw LLM output attached (for audit logging). Map this to a 422 response in the API layer with a user-friendly message ("The AI could not generate a valid query for your request").

### IMP-015 · CORRECTNESS · Enforce `extra="forbid"` on Critical Pydantic Models

**Current state:** Pydantic models like `ExecutionContext` silently ignore extra fields (default behavior). This masks BUG-007.
**Suggestion:** Add `model_config = ConfigDict(extra="forbid")` to `ExecutionContext`, `UserIntent`, `PromptEnvelope`, and `ExecutableQuery`. This turns silent extra-field pass-through into loud `ValidationError` at the earliest possible point.

### IMP-016 · CORRECTNESS · `RegistryLoader` Hot-Reload Across Workers

**File:** `app/steward/loader.py`, `app/main.py`
**Current state:** `RegistryLoader.load_active_schema()` is called once at startup and stored in `app.state.registry`. When a new schema version is activated, only the worker that handles the activation HTTP request updates its in-memory registry. Other workers continue using the old schema until restarted.
**Suggestion:** Implement a reload mechanism: either (a) use a Redis pub/sub channel to broadcast schema invalidation to all workers, or (b) add a version stamp to each compilation request and check it on every compile call, reloading lazily if the active version has changed.

### IMP-017 · PERFORMANCE · Add Connection Pooling Configuration to `asyncpg`

**File:** `app/main.py`
**Current state:** `create_async_engine` is called with no pool configuration. SQLAlchemy defaults (`pool_size=5`, `max_overflow=10`) may be too small or too large depending on load.
**Suggestion:** Expose pool configuration via environment variables:


### IMP-018 · SECURITY · Prevent `physical_target` Leakage via Audit Log

**File:** `app/audit/models.py`, `app/audit/logger.py`
**Current state:** `QueryAuditEvent` stores both `abstract_sql` and `physical_sql`. If audit logs are accessible to non-admin users (e.g., through an API endpoint that shows "your query history"), `physical_sql` would expose physical table/column names — breaking the abstract/physical separation guarantee.
**Suggestion:** Ensure any audit log API endpoint returns only `abstract_sql`, never `physical_sql`. Consider storing `physical_sql` in a separate, admin-only log stream. Add a note in `QueryAuditEvent` docstring marking `physical_sql` as sensitive.
