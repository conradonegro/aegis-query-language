## 1. System Overview

Aegis is a **semantic translation and security proxy** between natural language queries and PostgreSQL. It accepts a free-form user question, compiles it to parameterized SQL through an LLM, validates every node of the AST against an explicit allow-list, resolves abstract aliases to physical database targets, and executes the query against a least-privilege role. Users never write SQL directly, and the LLM never sees real table or column names.

### High-Level Request Flow

```
Client (NL question)
        │
        ▼
┌──────────────┐
│  app.api     │  POST /api/v1/query/generate   or   /execute
│  (FastAPI)   │
└──────┬───────┘
       │ UserIntent + session context
       ▼
┌──────────────────────────────────────────────────────────┐
│  app.compiler.CompilerEngine.compile()                   │
│                                                          │
│  1. RAG lookup      ──► app.rag.InMemoryVectorStore      │
│  2. Schema filter   ──► DeterministicSchemaFilter        │
│  3. Prompt build    ──► PromptBuilder (Jinja2)           │
│  4. LLM call        ──► LLMGateway (Ollama/OAI/…)       │
│  5. AST parse       ──► SQLParser (sqlglot/Postgres)     │
│  6. Safety check    ──► SafetyEngine (allow-list)        │
│  7. Translate       ──► DeterministicTranslator          │
└──────┬───────────────────────────────────────────────────┘
       │ ExecutableQuery (parameterized SQL + bound values)
       ▼
┌──────────────┐
│ app.execution│  asyncpg · SET LOCAL statement_timeout
│ ExecutionEngine
└──────┬───────┘
       │ QueryResult (columns + rows)
       ▼
┌──────────────┐
│  app.audit   │  fire-and-forget JSON audit log
└──────────────┘
       │
       ▼
 Client (JSON response)
```

## 2. Bounded Contexts & Import Rules

Module boundaries are enforced at CI time by `import-linter` (configured in `.importlinter`). Violating these rules fails `uv run lint-imports`.

| Context | Package | Rule |
|---------|---------|------|
| Steward | `app.steward` | May only be imported by `app.compiler`, `app.api`, `app.execution` |
| Compiler | `app.compiler` | May import `app.steward`; must NOT import `app.execution` |
| Execution | `app.execution` | May import `app.compiler`, `app.steward` |

### Shared Cross-Context Object

`RegistrySchema` (defined in `app.steward.models`) is the **only** object that crosses context boundaries. It is loaded once at startup by `RegistryLoader` and passed read-only into the compiler pipeline. No other steward internals are referenced outside `app.steward`.

## 3. Context: `app.steward` — Schema & Policy Definition

### Responsibility
Owns the canonical description of the database schema that the system can query. Decouples physical DB targets from what the LLM sees.

### Key Types

```
RegistrySchema
├── tables: dict[str, AbstractTableDef]
│   ├── alias: str                  # what the LLM sees (e.g. "orders")
│   ├── physical_target: str        # real table name (e.g. "public.ord_v2")
│   ├── description: str
│   └── columns: dict[str, AbstractColumnDef]
│       ├── alias: str
│       ├── physical_target: str
│       ├── data_type: str
│       ├── description: str
│       └── safety: SafetyClassification
│
├── relationships: list[AbstractRelationshipDef]
│   ├── from_table_alias, from_column_alias
│   └── to_table_alias, to_column_alias
│
└── tenant_id: str
```

### `SafetyClassification` (5 boolean flags per column)

| Flag | Controls |
|------|---------|
| `allowed_in_where` | Column may appear in WHERE predicates |
| `allowed_in_select` | Column may appear in SELECT list |
| `allowed_in_group_by` | Column may appear in GROUP BY |
| `aggregation_allowed` | Column may be wrapped in aggregate functions |
| `join_participation_allowed` | Column may be used as a JOIN key |

### `RegistryLoader`

`RegistryLoader.load_active_schema(session)` queries the latest `CompiledRegistryArtifact` (status = `active`), verifies SHA256 hash integrity of the JSON blob, verifies HMAC signature (using `compute_artifact_hmac_signature` / `verify_hmac_signature`), then hydrates into a `RegistrySchema`. This ensures registry artifacts cannot be silently tampered with in the database.

## 4. Context: `app.compiler` — Compilation Pipeline

### Responsibility
Transform a `UserIntent` (natural language) into a validated, parameterized `ExecutableQuery`. Never connects to a database. Never exposes physical table/column names to the LLM.

### `CompilerEngine.compile()` — Step-by-Step

#### Step 1: Session Context Reconstruction
```python
ctx = self.session_store.get(intent.session_id)
# injects prior FilteredSchema + chat history for follow-up queries
```
`SessionStore` uses Redis (multi-worker) with an in-memory LRU fallback (cap 10,000 sessions). TTL is configurable. Follow-up detection uses `DeterministicSchemaFilter.is_follow_up()` — token overlap with previous query's table scope.

#### Step 2: RAG Lookup
```python
rag_result = await self.rag_store.lookup(intent.query, tenant_id, top_k)
```
`InMemoryVectorStore` uses `difflib.SequenceMatcher` for fuzzy string matching against categorical column values. Returns a `RAGResult` containing column hints and an outcome (`NO_MATCH`, `SINGLE_HIGH_CONFIDENCE_MATCH`, `AMBIGUOUS_MATCH`). Threshold: 0.85. Hints are injected into the prompt.

#### Step 3: Schema Filter
```python
filtered = self.schema_filter.filter(intent, schema, rag_result)
```
`DeterministicSchemaFilter` tokenizes the user query and scores tables by token overlap against table/column aliases and descriptions. Expands the set by 1 degree via declared relationships. Returns a `FilteredSchema` (subset of `RegistrySchema`).

#### Step 4: Prompt Build
```python
envelope = self.prompt_builder.build(intent, filtered_schema, hints, history)
```
`PromptBuilder` renders `app/compiler/templates/system.jinja` via Jinja2. The template instructs the LLM to:
- Output **only** `{"sql": "..."}` JSON
- Use abstract table/column aliases only
- Refuse destructive operations by outputting `{"refused": true, "reason": "..."}`
- Obey SQL rules (no CTEs, no subqueries, no DDL/DML, `COUNT(*)` not `COUNT(column)`)

Chat history is truncated to the last 10 messages. **Critical invariant: `PromptEnvelope` must never include `physical_target` fields.**

#### Step 5: LLM Gateway Call
```python
llm_result = await self.llm_gateway.generate(envelope)
```
Returns `LLMResult(raw_text: str, token_usage: dict)`. All real gateways extract `parsed["sql"]` and return plain SQL. The mock gateway returns `{"sql": "..."}` JSON (inconsistency — see bug report).

#### Step 6: JSON Extraction & Refusal Check
```python
parsed = json.loads(llm_result.raw_text)  # or regex fallback
if parsed.get("refused"):
    raise LLMRefusalError(parsed["reason"])
```
The engine tries `json.loads`, then regex `\{.*\}` extraction, then treats raw text as SQL. This means the engine's refusal detection only works when the gateway returns full JSON — currently only the mock gateway.

#### Step 7: AST Parse
```python
validated_ast = self.parser.parse(abstract_sql)
```
`SQLParser` uses `sqlglot.parse()` with `dialect="postgres"`. Enforces exactly 1 statement. Returns `ValidatedAST`.

#### Step 8: Safety Engine
```python
self.safety_engine.validate(validated_ast)
```
`SafetyEngine` maintains:
- **DENY_LIST** (any of these = immediate reject): `Create`, `Drop`, `Insert`, `Update`, `Delete`, `Alter`, `Truncate`, `Merge`, `Copy`, `Grant`, `Revoke`, `Subquery`, `With` (CTEs), `Anonymous` (unknown functions), `Union`, `Intersect`, `Except`
- **ALLOW_LIST** (everything else must be in this set): `Select`, `From`, `Join`, `Where`, `Group`, `Having`, `Order`, `Limit`, `Column`, `Table`, `Star`, `Alias`, `EQ`, `NEQ`, `GT`, `GTE`, `LT`, `LTE`, `In`, `Like`, `ILike`, `Is`, `IsNull`, `Not`, `And`, `Or`, `Between`, `Literal`, `Cast`, `TryCast`, `Count`, `Sum`, `Avg`, `Min`, `Max`, `Paren`, `Add`, `Sub`, `Mul`, `Div`, standard date/time functions, etc.
- Cross-join and implicit join (comma-separated FROM list) are also rejected.

#### Step 9: Deterministic Translation
```python
executable = self.translator.translate(validated_ast, filtered_schema, ...)
```
`DeterministicTranslator` performs (in order):
1. **Alias resolution**: Maps abstract table aliases → physical targets (copy-on-write).
2. **Column validation**: Each column reference checked against `SafetyClassification`.
3. **Orphaned alias repair**: Detects and replaces dialect-specific expression aliases that sqlglot cannot re-resolve.
4. **JOIN graph validation**: Every JOIN must traverse a declared relationship in `RegistrySchema`; no ad-hoc JOINs.
5. **WHERE→HAVING relocation**: Moves aggregate predicates from WHERE to HAVING.
6. **EXTRACT/INTERVAL validation**: Ensures date parts are valid.
7. **Literal parameterization**: All string and numeric literals → `:p1`, `:p2`, … with `bound_parameters` dict.
8. **Row limit enforcement**: Injects `LIMIT :_row_limit` if not already present, up to configured max.
9. **Physical column resolution**: All column references resolved to `physical_table.physical_column`.

Output: `ExecutableQuery(sql: str, parameters: dict, abstract_sql: str, metadata: dict)`.

## 5. Context: `app.execution` — Database Execution

### Responsibility
The **only** layer permitted to open database connections. Executes raw parameterized SQL against PostgreSQL.

### `ExecutionEngine`

```python
async def execute(query: ExecutableQuery, context: ExecutionContext) -> QueryResult:
    async with self.engine.begin() as conn:
        await conn.execute(text("SET LOCAL statement_timeout = :t"), {"t": context.statement_timeout_ms})
        result = await conn.execute(text(query.sql), query.parameters)
    return QueryResult(columns=[...], rows=[...])
```

- Uses **SQLAlchemy async** engine backed by `asyncpg`.
- `SET LOCAL statement_timeout` scoped to each transaction — prevents runaway queries.
- `DB_URL_RUNTIME` is the dedicated least-privilege connection pool.
- Parameters from `ExecutableQuery.parameters` are passed as-is to the DB driver (no string interpolation — SQL injection prevention).

### `ExecutionContext`

```python
@dataclass
class ExecutionContext:
    tenant_id: str
    user_id: str
    statement_timeout_ms: int = 5000
```

## 6. Context: `app.audit` — Telemetry Sink

### Responsibility
Out-of-band logging of every query event. Must never block the API response or raise exceptions.

### `QueryAuditEvent` Fields

| Field | Type | Description |
|-------|------|-------------|
| `query_id` | UUID | Unique per request |
| `tenant_id` | str | Tenant |
| `nl_query` | str | Original natural language |
| `abstract_sql` | str | SQL before physical translation |
| `physical_sql` | str | SQL after translation |
| `latency_ms` | float | End-to-end compilation + execution latency |
| `token_usage` | dict | LLM token counts |
| `rag_outcome` | str | RAG match result |
| `rag_column_hints` | list | Columns injected by RAG |
| `safety_version` | str | Safety rule version stamp |

### `JSONAuditLogger`

Wraps `emit()` in `try/except Exception` to guarantee it never raises. Serializes the event to a JSON line via `json.dumps`.

### Cryptographic Chaining (`app/audit/chaining.py`)

Used for the WORM audit chain in the metadata layer:

- **`get_canonical_json(data)`** — sorts keys deterministically before serializing (prevents ordering-based evasion).
- **`compute_audit_row_hash(event, previous_hash)`** — SHA256 over canonical JSON + `previous_hash`. Creates an append-only chain; tampering with any row invalidates all subsequent hashes.
- **`compute_artifact_hmac_signature(artifact_json, secret_key)`** — HMAC-SHA256 of the compiled registry artifact.
- **`verify_hmac_signature(artifact_json, signature, secret_key)`** — Constant-time comparison via `hmac.compare_digest` (timing-safe).

## 7. Context: `app.rag` — Value Vector Store

### Responsibility
Pre-query semantic enrichment. Matches tokens in the user's question against a catalog of known categorical column values, injecting `PromptHints` to guide the LLM toward correct predicates.

### `InMemoryVectorStore`

```python
lookup(query: str, tenant_id: str, top_k: int = 3) -> RAGResult
```

Matching strategy (per value):
1. **Exact word match** — score 1.0
2. **Substring match** — score 0.9
3. **`difflib.SequenceMatcher` fuzzy ratio** — score = ratio (threshold 0.85)

Tenant isolation: values are keyed by `tenant_id`. No cross-tenant leakage.

Outcomes:
- `NO_MATCH` — no value above threshold
- `SINGLE_HIGH_CONFIDENCE_MATCH` — exactly one value above threshold per column
- `AMBIGUOUS_MATCH` — multiple matches (hint injected with all candidates)

### Session Management

Both `/generate` and `/execute` handlers participate in the same session lifecycle:

```
POST body: { session_id: str | null, ... }
    │
    ├─ if session_id null → CREATE new ChatSession in DB
    └─ if session_id given → LOAD existing ChatSession (SELECT FOR UPDATE)
         │
         ├─ compile(UserIntent, session_ctx)
         │       │
         │       └─ SessionStore.get(session_id) → prior FilteredSchema
         │
         └─ store result
              ├─ ChatMessage appended to DB (abstract SQL stored, not physical)
              └─ SessionStore.set(session_id, new_ctx)
```

**Key invariant**: `ChatMessage.abstract_sql` stores the pre-translation SQL. Physical SQL is never persisted to chat history.

### Schema Version Lifecycle

```
DRAFT → UNDER_REVIEW → APPROVED → COMPILED → ACTIVE
                                  └→ REJECTED
```

- Only one `ACTIVE` version may exist per tenant at a time.
- Transitions are validated (`VersionStatusUpdateRequest` with field-level rules).
- Every transition writes a WORM audit record (`MetadataAudit`) with SHA256 hash chaining.
- `POST /compile/{id}` compiles an `APPROVED` version into a signed `CompiledRegistryArtifact` and transitions it to `COMPILED`.

### `MetadataCompiler`

`MetadataCompiler.compile_version(session, version_id)`:
1. Loads all tables, columns, relationships for the version.
2. Builds a JSON artifact with full abstract→physical mapping.
3. Computes HMAC signature over the artifact JSON.
4. Writes `CompiledRegistryArtifact` + WORM audit row.
5. Returns the signed artifact.

## 10. Security Architecture

### Defense Layers (in order of application)

1. **Prompt hardening** — System prompt instructs LLM to refuse destructive intents; physical names never in prompt.
2. **AST allow-list** — Every sqlglot node type in the parsed query must be in the explicit allow-list; anything unknown = reject.
3. **JOIN graph validation** — Only joins along declared relationships are permitted; ad-hoc cross-entity joins blocked.
4. **SafetyClassification enforcement** — Per-column flag enforcement happens during physical translation; prevents column misuse even if AST passes.
5. **Literal parameterization** — All LLM-generated string/number values bound as DB driver parameters (`:p1`, `:p2`); no SQL string interpolation.
6. **Statement timeout** — `SET LOCAL statement_timeout` on every query prevents DoS via expensive queries.
7. **Role isolation** — Each operation class uses the minimally-privileged PostgreSQL role.
8. **WORM audit chain** — All schema changes cryptographically chained; PostgreSQL trigger blocks UPDATE/DELETE on audit table.
9. **Registry integrity** — Compiled schema artifact verified via SHA256 + HMAC on every load.

### Secrets Management (`app/vault.py`)

SECRETS_PROVIDER=env    → EnvFallbackProvider (dev only; blocked in ENVIRONMENT=production)

## 14. Key Invariants (Non-Negotiable)

1. `PromptEnvelope` → LLM: **zero physical targets** (no `physical_target` field values exposed).
2. `ExecutableQuery` output: **zero abstract aliases** (all resolved to physical targets before execution).
3. `ValidatedAST` is **immutable** — translator produces a copy, never mutates in-place.
4. All LLM-generated literals → **DB driver parameters** (`:p1`, `:p2`…), never string-interpolated.
5. `app.execution` is the **only** layer that opens database connections.
6. `app.audit.JSONAuditLogger` **never raises** — wrapped in broad `try/except Exception`.
7. `MetadataAudit` rows are **write-once** — PostgreSQL trigger raises on UPDATE/DELETE.
8. Only one `CompiledRegistryArtifact` with `status = active` per tenant at any time.
